## Частина 1. Аналіз помилок у `src/app/actions/transfer.ts`

Нижче наведено знайдені помилки, впорядковані за рівнем критичності (від найвищого до найнижчого).

### 1. Відсутність атомарності транзакції (Race Condition)
* **Що не так:** Зчитування балансу відправника, перевірка ліміту та безпосереднє оновлення балансів (списання й зарахування) відбуваються як окремі, не пов'язані між собою запити до бази даних.
* **Критичність:** **Critical**
* **Наслідки для грошей:** Якщо користувач ініціює кілька запитів на переказ одночасно (наприклад, швидким подвійним кліком або через скрипт), система може перевірити баланс паралельно для обох запитів і дозволити обидва. Це призведе до "подвійного списання" (double spending) та від'ємного балансу на рахунку, тобто фактичної втрати грошей банком/системою.

### 2. Відсутність авторизації відправника (Broken Object Level Authorization - BOLA)
* **Що не так:** Функція приймає параметр `fromAccountId`, але ніяк не перевіряє, чи належить цей рахунок авторизованому користувачу, який викликав Server Action. Ідентифікатор користувача з сесії (`auth()`) взагалі не використовується для перевірки прав власності.
* **Критичність:** **Critical**
* **Наслідки для грошей:** Будь-який автентифікований користувач системи може здійснювати перекази з будь-якого чужого рахунку на свій (або будь-який інший), просто підставивши чужий `fromAccountId` у запит. Це дозволяє повністю спустошити рахунки інших клієнтів.

### 3. Відсутність перевірки відповідності валют
* **Що не так:** Система дозволяє переказувати кошти між рахунками з різними валютами (наприклад, з USD на EUR) за курсом 1:1, оскільки в коді немає жодної перевірки чи конвертації.
* **Критичність:** **High**
* **Наслідки для грошей:** Користувач може створити рахунок у EUR та рахунок у USD, після чого переказати 100 USD на EUR-рахунок і отримати рівно 100 EUR. Це призведе до миттєвого викривлення балансів системи та фінансових збитків через різницю курсів.

### 4. Відсутність валідації суми переказу (Negative / Zero Transfer)
* **Що не так:** Немає перевірки на те, що сума переказу має бути строго додатною (`amount > 0`).
* **Критичність:** **High**
* **Наслідки для грошей:** Зловмисник може вказати від'ємну суму переказу (наприклад, `-100`). У такому разі формула оновлення балансу зробить відправнику `- (-100)` (тобто зарахує +100), а отримувачу `+ (-100)` (тобто спише 100). Це дозволяє несанкціоновано списувати гроші з чужих рахунків на свій.

### 5. Можливість переказу самому собі
* **Що не так:** Код не забороняє вказувати один і той самий рахунок як відправника (`fromAccountId`) та отримувача (`toAccountId`).
* **Критичність:** **Medium**
* **Наслідки для грошей:** Хоча це безпосередньо не веде до крадіжки коштів, такі транзакції засмічують базу даних фейковими записами й можуть викликати непередбачувану поведінку бізнес-логіки (наприклад, нарахування комісій або збої у звітності).

## Частина 3. Короткі питання

### Питання 1. Чому баланс краще зберігати не у форматі `Float`? Як би ти зберігав гроші?
Зберігання грошей у типі `Float` або `Double` (числа з рухомою комою) є грубою помилкою у фінансових системах через специфіку стандарту IEEE 754. Такі числа не можуть абсолютно точно представити деякі десяткові дроби (наприклад, `0.1 + 0.2` у JavaScript дасть `0.30000000000000004`). З накопиченням мільйонів транзакцій ці дрібні похибки перетворяться на цілком реальні відсутні або зайві долари.

**Як правильно зберігати гроші:**
1. **У мінімальних грошових одиницях (Cents / Копійки) як Integer / BigInteger:** Наприклад, $10.50 зберігається в базі даних як `1050` (ціле число). Це найпростіший та найпродуктивніший спосіб, який повністю усуває проблему дробів на рівні БД.
2. **Тип `Decimal` / `Numeric` на рівні БД:** Спеціальний тип даних фіксованої точності (наприклад, `Decimal(12, 2)` або `Decimal(18, 4)` для підтримки криптоактивів чи точних розрахунків). Prisma підтримує тип `Decimal` з коробки, що дозволяє працювати з грошима без втрати точності.

### Питання 2. Два перекази з одного рахунку відбуваються "одночасно". Що може піти не так і як це можна не допустити?
**Що може піти не так:**
Виникає класичний стан гонитви (Race Condition):
1. Транзакція А зчитує баланс рахунку (баланс = 100$).
2. Транзакція Б паралельно зчитує той самий баланс рахунку (баланс = 100$).
3. Транзакція А списує 80$ (перевірка `100 >= 80` успішна), оновлює баланс до 20$.
4. Транзакція Б списує 50$ (перевірка `100 >= 50` також успішна, бо вона зчитала баланс до оновлення), оновлює баланс до 50$ (або перезаписує результат транзакції А).
В результаті баланс стає або від'ємним (-30$), або відбувається втрата даних про списання однієї з транзакцій.

**Як запобігти:**
1. **Песимістичне блокування (Pessimistic Locking / `SELECT FOR UPDATE`):** Під час зчитування рахунку в межах транзакції рядок бази даних блокується для будь-яких інших операцій читання/запису до завершення поточної транзакції. У PostgreSQL це робиться через `SELECT ... FOR UPDATE`.
2. **Оптимістичне блокування (Optimistic Locking):** Додавання колонки `version` або `updatedAt` до моделі рахунку. Під час оновлення перевіряється, чи версія не змінилася: `UPDATE Account SET balance = balance - X, version = version + 1 WHERE id = Y AND version = current_version`. Якщо рядок вже було змінено іншим запитом, транзакція відкочується.
3. **Рівень ізоляції транзакцій `Serializable`:** Найсуворіший рівень ізоляції в БД, за якого СУБД автоматично стежить за тим, щоб транзакції виконувалися так, ніби вони йдуть строго одна за одною.

### Питання 3. Як би ти покрив функцію переказу тестами? Які 3–4 випадки перевірив би в першу чергу?
Для покриття цієї бізнес-логіки я б написав інтеграційні тести (з використанням тестової бази даних), оскільки тут критично перевірити взаємодію з БД та роботу транзакцій.

**Першочергові тест-кейси:**
1. **Успішний переказ (Happy Path):** Перевірка, що за правильних умов (достатньо коштів, однакова валюта, правильний власник) баланс відправника зменшився, баланс отримувача збільшився на точну суму, а в базі створився запис про успішну транзакцію.
2. **Тест авторизації (Security/Auth Check):** Спроба здійснити переказ із рахунку, який не належить поточному залогіненому користувачу. Очікуваний результат: помилка авторизації, баланси не змінюються.
3. **Тест недостатньої кількості коштів (Insufficient Funds):** Спроба переказати суму, яка є більшою за поточний баланс відправника. Очікуваний результат: помилка валідації балансу, транзакція скасовується, баланси залишаються незмінними.
4. **Тест несумісності валют (Currency Mismatch):** Спроба переказу між рахунками з різними валютами (USD та EUR). Очікуваний результат: помилка валідації валют, кошти не списуються.
5. **Тест на стан гонитви (Race Condition / Concurrency):** Одночасний виклик 5 паралельних запитів на списання суми, яка в сумі перевищує ліміт (наприклад, 5 запитів по 30$ при балансі 100$). Очікуваний результат: лише 3 транзакції проходять успішно, баланс стає рівним 10$, інші 2 транзакції завершуються з помилкою.
